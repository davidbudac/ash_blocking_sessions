ASH · DBA_HIST_ACTIVE_SESS_HISTORY

Who blocked whom, for how long, and why.

When an app freezes on database locks, native V$ views only show now. This tool reads the blocking samples Oracle already kept in DBA_HIST_ACTIVE_SESS_HISTORY and rebuilds the whole incident into one self-contained HTML report — a per-session timeline and a click-through blocking tree for any 10-second ASH sample.

The problem

The lock cleared. The question didn't.

By the time you're paged, the blocking is often already over. You need to answer, from history: which session was the root blocker, how many sessions it held up and for how long, and — the decisive part — whether it was running SQL or just sitting on an uncommitted transaction. AWR keeps the samples; reconstructing the chains by hand from DBA_HIST_* is the slow part.

V$ views

Show only now. Live blocking views are empty once the incident passes — nothing left to inspect.

ASH history

Keeps the evidence at a ~10-second heartbeat, but the chains are scattered across samples and instances.

This tool

Rebuilds the incident — every chain, every waiter, every root — and lays it out so you read the cause in seconds.

The report — three linked views

Verdict, timeline, and the exact chain shape.

Each view answers a different question, and they're wired together: click a cell in the timeline to open that instant's blocking tree; step sample by sample to watch chains form and collapse.

Production stance

Nothing is installed in your database.

The whole tool is one read-only PL/SQL block plus a shell script that runs on your machine. Here is exactly what touches the database — and everything that never does.

runs on the DB A single read-only query

  • SELECTs only against DBA_HIST_*, CDB_USERS, CDB_OBJECTS, V$DATABASE.
  • Plus session-scoped ALTER SESSION for NLS formats — nothing that outlives your session.
  • Reads every container in the CDB; the two JSON payloads are printed to stdout.
  • Works unchanged against a remote database over EZConnect / TNS.

never happens No footprint, ever

  • No tables, views, or directory objects — no database objects of any kind.
  • No DDL and no DML — the tool cannot modify anything.
  • No agent, app server, or scheduled job left running.
  • No server-side files written — the HTML is assembled on your machine.
Least privilege A common user with CREATE SESSION+SELECT_CATALOG_ROLE — or connect asSYSDBA. Nothing else.

How it works

SQL in, one HTML file out — assembled on your side.

No round trip through a server. sqlplus prints JSON; a POSIX shell script splices it into a template locally.

on DB
01
sqlplus
runs build_report.sql read-only
→
on DB
02
SELECT DBA_HIST ASH
self-joined to enrich each waiter with its blocker; + CDB_USERS / CDB_OBJECTS
→
on DB
03
JSON to stdout
two payloads (meta + data) between marker lines
→
on you
04
awk splice
run_report.sh merges JSON into template.html
→
on you
05
reports/*.html
one self-contained file, charts via ECharts

Nothing is written on the database server. Because the report is assembled client-side from query output, the machine running the scripts and the database can be entirely different — the tool works the same against a remote DB, and the report always lands in your local reports/.

Requirements — stated honestly

What you need before you run it.

01

Oracle 12.2+ with Diagnostics Pack

The report reads DBA_HIST_*, which is licensed under the Diagnostics Pack.

02

sqlplus on PATH

Set the Oracle environment first — e.g. . oraenv. No other client tooling.

03

A window inside AWR retention

The incident window must still be in AWR history (default retention ≈ 8 days).

04

A POSIX shell

The scripts are plain sh — they run on AIX ksh, ksh93, and bash unchanged.

Quick start

Clone, point it at a window, open the report.

~/ash_blocking_sessions 
$ git clone https://github.com/davidbudac/ash_blocking_sessions.git
$ cd ash_blocking_sessions

# last 2h, all containers, / as sysdba
$ ./run_report.sh

# a specific incident window
$ ./run_report.sh "/ as sysdba" 2026-07-01T13:30:00 2026-07-01T15:30:00

# open the newest report
$ open "$(ls -1t reports/*.html | head -n1)"

Times are ISO YYYY-MM-DDTHH24:MI:SS (uppercase T, no spaces). Arg 1 is the sqlplus connect string; a named user must be a common user in CDB$ROOT.
